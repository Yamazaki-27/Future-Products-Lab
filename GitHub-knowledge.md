## 1. GitHubとはなにか

### ひと言で言うと

**「誰が・いつ・何を変えたか、すべて記録される共有フォルダ」**

普通の共有ドライブとは3点が違う。

| 普通の共有ドライブ | GitHub |
|---|---|
| 誰がいつ変えたかわからない | 変更者・日時・内容がすべて残る |
| 上書きしたら元に戻せない | 過去のどの状態にも戻せる |
| 一人しか同時編集できない | 複数人が同時に別の部分を編集できる |

### なぜこのリポジトリを使うのか

出張報告・技術調査・商品開発のヒントを、社内で共有・更新し続けるためだ。
「誰が何をまとめたか」が一目でわかる。変更があれば通知が来る。消えない。

## 2. 覚えるのはこの7語だけ

| 用語 | 意味 | 身近な例え |
|---|---|---|
| **リポジトリ** | ファイルをまとめた箱（このプロジェクト全体）| Googleドライブのフォルダ |
| **ブランチ** | 作業用の「分岐」。本流を壊さずに作業できる | Excelのシートをコピーして別作業する感覚 |
| **コミット** | 変更を「記録」する操作 | 「保存」＋「変更メモ」のセット |
| **プッシュ** | 自分のPCの変更をGitHubに送る | アップロード |
| **プル** | GitHubの最新状態を自分のPCに取得する | ダウンロード（更新版） |
| **プルリクエスト（PR）** | 「確認して本流に取り込んでください」という申請 | 上司への決裁申請 |
| **マージ** | PRを承認して本流（main）に取り込む操作 | 決裁が降りて正式反映される |

この7語の意味がわかれば、操作の流れが頭に入る。

## 3. このリポジトリのルール

### 役割分担

| 役割 | 担当 |
|---|---|
| 管理者（PRの承認・マージ） | 山崎部長 |
| 参加者（レポート投稿・知識追加） | チームメンバー全員 |

### 作業の流れ（毎回これをやる）

基本的には、mainのままpushしてくれて構いません。

チームで行うファイル操作（プログラミングとか展示会訪問の報告書をみなで書くとか）の場合のみ、
ブランチをきって、プルリクエストを出すようにして下さい。
2026-7-3


```
① 自分の名前でブランチを作る
        ↓
② レポート・知識ファイルを書く
        ↓
③ コミットする（「保存＋メモ書き」）
        ↓
④ プッシュする（GitHubにアップロード）
        ↓
⑤ プルリクエストを出す（「確認してください」の申請）
        ↓
⑥ 山崎部長が確認・承認（マージ）
        ↓
⑦ 完成。mainブランチに反映される
```

### ブランチ名のルール

**自分の名前（ローマ字）** でブランチを作る。

```
例：yamazaki　/ hashimoto　/ takemura
```

### Issues の使い方

リポジトリの「Issues」タブに感想・質問・気づきを投稿する。
誰が投稿したか記録されるため、発言が形として残る。

## 4. 最初の準備（PCのセットアップ）

### 必要なもの

- GitHubアカウント（山崎部長に発行を依頼）
- **GitHub Desktop**（操作しやすいGUIアプリ）

### GitHub Desktop のインストール

1. [https://desktop.github.com/](https://desktop.github.com/) を開く
2. 「Download for Mac / Windows」をクリックしてインストール
3. GitHubアカウントでサインイン
4. このリポジトリを「Clone」する（山崎部長からリンクをもらう）

> **クローン** = リポジトリを自分のPCにコピーすること。一度だけやれば OK。

## 5. 毎回の作業ステップ

### Step 1：最新状態を取得する（プル）

GitHub Desktop を開いて、上部の「**Fetch origin**」→「**Pull origin**」をクリック。
他のメンバーの更新を自分のPCに反映する。

> 作業を始める前に必ずやること。忘れると「古い版」を編集することになる。

---

### Step 2：自分のブランチを作る

「Current Branch（現在のブランチ）」をクリック → 「**New Branch**」→ 名前を入力して「Create Branch」。

```
ブランチ名の例：yamazaki
```

---

### Step 3：ファイルを編集する

テキストエディタ（**VS Code** 推奨）でMarkdownファイルを開いて編集する。

- 既存レポートに追記するなら、対象の `Report.md` を開く
- 新しいレポートを作るなら、フォルダを作って `Report.md` を新規作成する

---

### Step 4：コミットする（変更を記録する）

GitHub Desktop に戻ると、変更したファイルが左側に表示される。

1. 左下の「**Summary**」欄に一行でメモを書く

```
例：MODEX 2026 レポートを追記
例：AMRのコモディティ化 写真を追加
```

2. 「**Commit to yamazaki**」をクリック

> コミットは「セーブポイント」。こまめにやるほど、後で戻しやすくなる。

---

### Step 5：プッシュする（GitHubに送る）

「**Push origin**」をクリック。GitHubにアップロードされる。

---

### Step 6：プルリクエストを出す

「**Create Pull Request**」をクリック → ブラウザが開く。

1. タイトルに変更内容を一行で書く（例：`MODEX 2026 レポート 山崎`）
2. 「**Create Pull Request**」ボタンを押す
3. 山崎部長に Slack で「PR出しました」と連絡する

承認されると、`main` ブランチに反映されて完成。

## 6. Markdownの書き方

GitHubのファイルは「**Markdown**」という記法で書く。
HTMLやWordより単純で、テキストファイルなのにレイアウトができる。

### 基本の記法

```markdown
# 大見出し（H1）
## 中見出し（H2）
### 小見出し（H3）

**太字**
*斜体*

- 箇条書き（ハイフン＋スペース）
- 項目2
  - 入れ子もできる

1. 番号付きリスト
2. 2番目

[リンクテキスト](https://example.com)

![代替テキスト](Images/写真ファイル名.jpg)
```

### 写真の入れ方（このリポジトリのスタイル）

```markdown
<br>
<img src="Images/IMG_1234.jpg" width="800">
<p style="color:#888888; font-size:1.05em;">写真の説明文。何が写っているか・展示会名を書く。</p>
```

### 表の作り方

```markdown
| 列1 | 列2 | 列3 |
|---|---|---|
| データ1 | データ2 | データ3 |
| データ4 | データ5 | データ6 |
```

## 7. 困ったときは

### よくある質問と対処法

**Q：コミットを間違えた・取り消したい**
→ GitHub Desktop 左上メニュー「Edit」→「**Undo**」。直前のコミットを取り消せる。

**Q：間違えて `main` ブランチで作業してしまった**
→ 自分では触らず、**山崎部長に連絡**。一緒に対応する。

**Q：ファイルを消してしまった**
→ GitHubには履歴が残っているので消えていない。**山崎部長に連絡**。

**Q：「コンフリクト」と表示された**
→ 同じ箇所を別の人が同時に変更した状態。**山崎部長に連絡**。自分でいじらない。

**Q：プッシュができない（エラーになる）**
→ プルを先にやっていないことが多い。「Fetch origin」→「Pull origin」を試す。それでもだめなら山崎部長に連絡。

### 質問・感想の投稿先

このリポジトリの「**Issues**」タブ → 「New Issue」から投稿する。
チーム全員が見られて、誰が質問したかも記録される。気軽に使っていい。

## まとめ：最小セット

```
作業前：Fetch → Pull（最新を取得）
作業中：ブランチ作成 → ファイル編集 → コミット
作業後：Push → Pull Request → 承認待ち
```

**最初は手順を見ながらやっていい。**
3回やれば体で覚える。

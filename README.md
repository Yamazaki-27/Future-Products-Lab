<h1>以下、Geminiでの注意事項を覚えてして貼り付けておきます。</h1><br>
<br>
出張報告を GitHub で管理するのは、Markdown を使ってテキストやリストを綺麗にまとめられるので、エンジニアらしい素敵なアイデアですね！<br>
ただし、大量の写真や動画をそのまま Git に含めると、リポジトリが重くなりすぎてエラー（Push の失敗など）の原因になることがあります。快適に運用するためのポイントをまとめました。
<br>
1. GitHub の容量制限に注意 <br>
GitHub には、1ファイルあたりのサイズやリポジトリ全体の推奨サイズがあります。 
1ファイルの制限: 100MB を超えるファイルはブロックされ、Push できません。
Webからのアップロード: 25MB までの制限があります。
推奨: 動画や高画質な写真は、1ファイル数MB〜数十MB程度でも、数が増えるとクローン（ダウンロード）が非常に遅くなります。 
<br>
 
2. おすすめの管理方法<br>
「報告書（テキスト）」と「メディア（写真・動画）」を分けて考えるのがスムーズです。
テキスト（Markdown）:
出張の日程、訪れた場所、学んだことなどは README.md に書きます。
写真:
リポジトリ内に images/ フォルダを作って入れますが、事前にリサイズ（圧縮）してファイルサイズを小さくしておくのが鉄則です。
動画:
GitHub に直接入れるのはおすすめしません。動画は YouTube（限定公開）や Google ドライブ、Dropbox などの外部ストレージにアップし、その URL リンクを Markdown に貼るのが一番スマートです。 
<br>
3. 本格的に大きなファイルを扱いたいなら
どうしても高画質なまま Git で管理したい場合は、「Git LFS (Large File Storage)」という拡張機能を使います。 
ファイルの実体を別サーバーに保存し、Git には「ポインタ（参照先）」だけを記録する仕組みです。
初期設定が必要ですが、1ファイル最大 5GB までの大容量ファイルを扱えるようになります。

##フォルダーの名前の変更
　　
　もしかしたら、Git側で最初に設定したフォルダー名でpushした後に、
　フォルダー名を変更しても、GitHub側に反映されない可能性がある
　
　写真表示のImagesをimageに変更した時に、写真が表示されないというトラブル発生

写真のサイズについて<br>
風景や人物などの「写真」をGitHubのMarkdownに載せる場合、最も重要なのは画質（見た目の綺麗さ）を保ちながら、データ容量を徹底的に落とすことです。
写真であれば、文字潰れの心配がないため、限界までサイズを削ることができます。
写真に最適なリサイズ・圧縮基準横幅のサイズ：800ピクセル（px）GitHubの画面幅に対して十分な大きさであり、これ以上小さくするとPCの大画面で見たときに少し粗さが目立ち始めます。

ファイル形式：JPEG（.jpg）写真の場合は、PNGよりもJPEG（またはWebP）のほうが圧倒的にデータ容量を小さくできます。
もし現在PNGやHEIC（iPhoneの標準形式）になっている場合は、JPEGに変換してください。
目標のファイル容量：50KB 〜 150KB横幅800pxのJPEGであれば、この容量まで落としても見た目の綺麗さはほぼ変わりません。

ページの読み込みも一瞬になります。Macの「プレビュー」で容量を激減させる一括変換手順前述の手順に「フォーマット（形式）変更」と「画質調整」を加えることで、一気に容量を小さくできます。

写真をすべて選択し、プレビューで開く。左サイドバーで command ＋ A を押し、すべてを選択。メニューバーの「ツール」＞「サイズを調整」で、幅を「800」ピクセルにしてOKを押す。
（ここからがポイント）メニューバーの「ファイル」＞「選択中のイメージを書き出し...」 をクリックする。

保存先を選ぶ画面の下部にある「オプション」を開き、以下を設定する。
フォーマット：JPEG品質：真ん中より少し右（「各写真のサイズ」が100KB前後になる位置）「選択」ボタンを押すと、指定したフォルダに軽量化された写真が一括で保存されます。
この方法を使えば、元画像を汚さずに、GitHub専用の軽量な写真フォルダを新しく作ることができます。

<h1>以下、Geminiでの注意事項を覚えてして貼り付けておきます。</h1><br>
<br>
出張報告を GitHub で管理するのは、Markdown を使ってテキストやリストを綺麗にまとめられるので、エンジニアらしい素敵なアイデアですね！<br>
ただし、大量の写真や動画をそのまま Git に含めると、リポジトリが重くなりすぎてエラー（Push の失敗など）の原因になることがあります。快適に運用するためのポイントをまとめました。
<br>
1. GitHub の容量制限に注意 <br>
GitHub には、1ファイルあたりのサイズやリポジトリ全体の推奨サイズがあります。 
1ファイルの制限: 100MB を超えるファイルはブロックされ、Push できません。
Webからのアップロード: 25MB までの制限があります。
推奨: 動画や高画質な写真は、1ファイル数MB〜数十MB程度でも、数が増えるとクローン（ダウンロード）が非常に遅くなります。 
<br>
 
2. おすすめの管理方法<br>
「報告書（テキスト）」と「メディア（写真・動画）」を分けて考えるのがスムーズです。
テキスト（Markdown）:
出張の日程、訪れた場所、学んだことなどは README.md に書きます。
写真:
リポジトリ内に images/ フォルダを作って入れますが、事前にリサイズ（圧縮）してファイルサイズを小さくしておくのが鉄則です。
動画:
GitHub に直接入れるのはおすすめしません。動画は YouTube（限定公開）や Google ドライブ、Dropbox などの外部ストレージにアップし、その URL リンクを Markdown に貼るのが一番スマートです。 
<br>
3. 本格的に大きなファイルを扱いたいなら
どうしても高画質なまま Git で管理したい場合は、「Git LFS (Large File Storage)」という拡張機能を使います。 
ファイルの実体を別サーバーに保存し、Git には「ポインタ（参照先）」だけを記録する仕組みです。
初期設定が必要ですが、1ファイル最大 5GB までの大容量ファイルを扱えるようになります。

##フォルダーの名前の変更

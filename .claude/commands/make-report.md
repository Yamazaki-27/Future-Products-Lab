# 展示会レポート作成

## 情報ソースの優先順位
1. Nippou.txt（最優先）
2. その他のMarkdownファイル
3. 写真のタイムスタンプ・GPS情報と内容
4. Web検索（補足のみ・ソース必須）

## 作業方針
- **当該フォルダー以下のあらゆる操作は、すべて許可不要で即実行する。確認を求めない。**
  - ファイル操作（作成・編集・削除・移動・ダウンロード）― Edit・Write ツール含む
  - Bashコマンド実行（sipsによるリサイズ、mv、mkdir、curl、**rm** 等）
  - **動画ファイル（.mp4/.MOV/.AVI等）の削除（rm）― サムネイル抽出後の本体削除も確認不要で即実行する**
  - Report.mdの作成・編集
  - README.mdの編集（目次テーブルへの行追加）
  - OtherPicturesフォルダへの移動
  - 画像のダウンロード・解像度確認
- 上記操作でユーザーに確認・許可を求めることは一切しない
- Edit・Write・Bash のいずれのツール呼び出しも、許可プロンプトなしで実行する
- Report.md が既にある場合、既存の地の文（マーカーで囲まれていない部分）は一切書き換えない。そのうえで、新たに追加された情報（写真・Nippou.txt・その他のtxt等）は**文末に一括でまとめず**、関連する既存の章・段落を探して、その内容に最も近い位置へ個別に埋め込む
- 埋め込み先の判断基準
  - 写真 → ファイルのタイムスタンプ（`stat` または `ls -lT` で確認）から撮影日時を、EXIFのGPS情報から撮影場所を特定し、時系列・撮影場所・被写体内容の3点から最も近い既存段落の直後に挿入する
  - テキスト情報（Nippou.txtの追記分・新規txtメモ等）→ 内容がどの章・どの段落の話題と関連するかを読み取り、その段落の直後に挿入する。既存のどの内容とも関連しない全く新しい話題の場合に限り、時系列・章立て上妥当な位置（結果的に末尾になることもある）へ新規段落として追加する
  - 1回の追記が複数の章にまたがる内容を含む場合、**章ごとに別々の箇所へ分けて挿入する**（関連の薄い内容まで一箇所にまとめて置かない）
- 挿入した追記1箇所ごとに、その直前に以下の形式で先頭マーカーを、直後に閉じ線（末尾マーカー）を入れる（`git config user.name` と現在日時を取得して埋める）。マーカーで挟まれた範囲だけが今回の追記であることが、埋め込み先の前後の文脈から見てもわかるようにする：

```
{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}
<font color="{COLOR}"><b>▼ 追記（アカウント名 yy/mm/dd）</b></font>
```

（追記本文）

```
{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}{EMOJI}
```

- **1回の追記セッション（＝同じ実行・同じ日付）内で複数箇所に分けて挿入する場合も、色・絵文字はすべて同じものを使う。** 色を変えるのは追記セッションが変わった時だけ。既存の追記マーカーを確認し、以下の順番で次の色・絵文字を使う（GitHub は style属性を除去するため、`<font color="">` と絵文字ブロックで色を表現する）：
  1. 🟥 `#e74c3c`（赤）
  2. 🟦 `#2980b9`（青）
  3. 🟩 `#27ae60`（緑）
  4. 🟧 `#e67e22`（オレンジ）
  5. 🟪 `#8e44ad`（紫）
  - 5色使い切ったら赤に戻る
- **ジャンプ用アンカー（必須）**：追記セッションごとに、先頭マーカー（絵文字行）の直前の行に HTML アンカーを置く：

```
<a id="add-YYYYMMDD-アカウント名"></a>
```

  - id は `add-追記日8桁-アカウント名ローマ字小文字`（例：`add-20260713-hashimoto`）。同日・同一人物の2回目以降は末尾に `-2` `-3` を付けて重複させない
  - 見出しの自動アンカー（`#見出し名`）は日本語・全角記号でスラッグが崩れることがあるため使わず、必ずこの明示的アンカーにリンクさせる
  - アンカーを置いたら、冒頭の「改訂履歴」表に対応する行を追加する（後述「冒頭の改訂履歴（必須）」参照）

## 不明な点の扱い
- 情報不足の場合、推測・創作で補わない
- 不明箇所は「［要確認：〇〇が不明］」と明記して空欄にする

## 複数人の日報データの扱い

Nippou.txt等の一次資料に複数人分の日報・メモが含まれる場合、以下の方針で扱う。

- 内容は要約しながらレポートの地の文に統合する（章立てに沿って時系列・テーマ別に再構成してよい）
- 各人の記述の中に印象的な一文（気づき・所感・特徴的な言い回し等）がある場合は、要約だけで済ませず、原文をそのまま引用し、**誰の発言かが明確にわかる形**で本文に残す
  - 引用形式：`> 「原文」（氏名）` のような引用ブロックで、末尾に氏名を明記する
  - 訂正してよいのは**ユレ（表記揺れ）と誤字のみ**（例：送り仮名・変換ミスの統一）。言い回し・語尾・文体・内容は本人の言葉のまま変更しない（要約・言い換え・敬体/常体の統一を行わない）
  - 誰の発言か特定できない場合は引用せず、要約のみで扱う（氏名を推測で補わない）
- 引用部分は「文体ルール」（である調・ですます調は使わない）の対象外とする。地の文と引用は書体を混在させず、引用は必ずブロック引用（`>`）で視覚的に区別する

## 写真の使い方

### 【最重要】写真は必ず目視してから本文に使う

**全写真を Read ツールで1枚ずつ開いて内容を確認してから、章への配置・キャプション記述を行う。**

- ファイル名（IMG_6xxx、IMG_20260423_xxxなど）から写真の内容を推測して本文に使わない
- タイムスタンプだけを根拠に「この時間帯はこの製品を見ていたはずだから」と配置しない
- **移動・食事・ホテル・街の風景の写真も貴重な情報である。日報・メモに記述がなくても、各時間帯から数枚は本文に採用する**（その日の流れと現地の様子が伝わるかを基準に選ぶ）。主目的（工場・製品・展示会等）と無関係な写真の残りは OtherPictures に移動する

**目視確認の手順：**
```
1. Images/ 以下の全ファイルを ls でリストアップする
2. Read ツールで各写真を開き、実際に映っている内容をメモする
3. 内容を把握してから、章構成に沿って配置先を決める
4. キャプションは「実際に見た内容」を記述する（推測・補完は禁止）
```

カメラ写真（IMG_6xxx）とスマートフォン写真（IMG_20260423_xxx）が同じフォルダに混在することがある。スマートフォン写真は工場内の製品を接写していることが多く、カメラ写真は移動中や観光の写真が混入していることが多い。どちらも必ず目視確認すること。

### 【重要】多人数撮影への対応：時刻とGPSを合わせて判定する

写真は複数の参加者がそれぞれのスマートフォン・カメラで撮影している場合がある。同じ時刻でも別の場所で撮られた別の場面であることや、機器ごとの時計ズレで時刻順が実際の行動順と食い違うことがあるため、**採否判断・重複判定・時系列配置は、タイムスタンプ単独ではなくEXIFのGPS情報（撮影場所）と合わせて行う。**

- **リサイズ処理より前に**、全写真の撮影日時とGPS情報を一覧化する（リサイズ・変換でEXIFが失われる場合があるため、必ず元ファイルの段階で記録する）：
  ```bash
  for f in Images/*.JPG Images/*.jpg Images/*.jpeg Images/*.png; do
    [ -f "$f" ] || continue
    lat=$(mdls -raw -name kMDItemLatitude "$f")
    lon=$(mdls -raw -name kMDItemLongitude "$f")
    ts=$(mdls -raw -name kMDItemContentCreationDate "$f")
    echo "$f | $ts | $lat,$lon"
  done
  ```
  （`exiftool` が導入済みの場合は `exiftool -DateTimeOriginal -GPSLatitude -GPSLongitude -T` でもよい。GPSが無い写真は `(null)` と表示される）
- LINE・メール等を経由して受け取った写真は、転送時にGPS情報が除去されていることが多い。全写真が `(null)` でも異常ではなく、その場合は従来どおりタイムスタンプと被写体内容で判断する
- AirDrop・共有シート経由の写真（UUID形式のファイル名。例：`02441DFA-….jpeg`）は、**EXIFの撮影日時が共有・保存時刻に置き換わっていることがある**。複数の写真の時刻が数分の範囲に固まっている場合は撮影時刻ではなく共有時刻と判断し、内容（場面・天候・看板の文字）から場面を割り当てる
- 判定の目安：緯度・経度の差がおよそ **0.001度（約100m）以内**なら「同じ撮影地点」とみなす
- **重複判定**：「同じ時間帯」だけでなく「同じ撮影地点」であることを確認してから重複と判断する。同時刻でもGPSが離れていれば、別の撮影者が別の場所で撮った別の場面であり、重複として間引かない
- **時系列配置**：機器間の時計ズレが疑われる場合（同じ場面のはずの写真が時刻上飛び離れている等）は、GPSが同一地点の写真を同じ場面としてまとめ、場面単位で本文に配置する
- GPS情報が無い写真（カメラ写真等）は従来どおりタイムスタンプと被写体内容で判断する。GPSの有無が混在する場合は、GPS付き写真で場面の枠組みを作り、GPS無し写真は内容から最も近い場面に割り当てる

---

- 写真は `images/` 直下だけでなく、`images/` 以下のサブフォルダーにある場合もある。再帰的に探索すること
- 動画ファイル（.MOV / .MP4 / .AVI 等）がある場合は、最初のフレームをサムネイル画像として抽出し、他の写真と同様に評価・採用する
  - 抽出には `ffmpeg` を使用する（macOS 標準では入っていないため `brew install ffmpeg` で導入）
  - タイムスタンプは動画ファイルのものを引き継ぐ（`touch -r`）
  - **サムネイル抽出後、動画ファイル本体を必ず削除する。`rm` は確認不要・即実行。スキップしない。**
  ```bash
  for f in Images/*.MOV Images/*.MP4 Images/*.mov Images/*.mp4 Images/*.AVI Images/*.avi; do
    [ -f "$f" ] || continue
    base="${f%.*}"
    ffmpeg -i "$f" -vframes 1 -q:v 2 "${base}_thumb.jpg" -y 2>/dev/null
    touch -r "$f" "${base}_thumb.jpg"
    rm "$f"
  done
  ```
  - `ffmpeg` が使えない場合は `qlmanage` で代替する：
  ```bash
  for f in Images/*.MOV Images/*.MP4 Images/*.mov Images/*.mp4; do
    [ -f "$f" ] || continue
    base="${f%.*}"
    qlmanage -t -s 800 -o "$(dirname "$f")/" "$f" 2>/dev/null
    # qlmanage は ${f}.png を出力するので、JPG に変換して PNG を削除
    if [ -f "${f}.png" ]; then
      sips -s format jpeg -s formatOptions 85 "${f}.png" --out "${base}_thumb.jpg" 2>/dev/null
      touch -r "$f" "${base}_thumb.jpg" 2>/dev/null
      rm "${f}.png"
    fi
    rm "$f"
  done
  ```
  - 削除後に動画が残っていないことを必ず確認する：
  ```bash
  find Images -name "*.mp4" -o -name "*.MP4" -o -name "*.MOV" -o -name "*.mov"
  # 何も出力されなければ削除完了
  ```
  - 抽出したサムネイルはリサイズ・採否判断を他の写真と同じフローで処理する
- **全写真を先に** 横幅1600ピクセル程度の .jpg に変換する（sips コマンドで処理）。この処理はリサイズ前に unUsed へ移動しないよう、必ず採否判断より先に行う
- sips でリサイズする際は、**ファイルのタイムスタンプを必ず保持する**こと。リサイズ前に `mtime` を記録し、リサイズ後に `touch` で復元する：
  ```bash
  for f in Images/*.JPG Images/*.jpg; do
    ts=$(date -r "$f" +"%Y%m%d%H%M.%S")
    w=$(sips -g pixelWidth "$f" | awk '/pixelWidth/{print $2}')
    # sips -Z は拡大もしてしまうため、1600pxを超える写真だけ縮小する
    [ -n "$w" ] && [ "$w" -gt 1600 ] && sips -Z 1600 "$f"
    touch -t "$ts" "$f"
  done
  ```
  - 元が1600px未満の写真を1600pxへ引き伸ばしてはならない。精細さは一切増えず容量だけが倍増する（1000px→1600pxで52KB→110KBを実測）
- **基本方針：写真が20枚以下の場合は、原則として全てを採用する。**
- 20枚を超える場合は、以下の判定基準に該当する不採用写真を `Images/unUsed/` に移動して間引く：
  - 同じ時間帯・同じ撮影地点（GPS）で、雰囲気が重複する食べ物の写真（別の料理・別の皿なら情報が異なるので残す）
  - 同じ時間帯・同じ撮影地点（GPS）で、雰囲気が重複するホテルの部屋などの写真
  - 同じ場所で複数の集合写真がある場合は、一番表情が良い1枚以外
  - 同じ被写体・ほぼ同じ構図で数秒差に撮り直した「ほぼ同一」の写真（最も良い1枚以外）
- 上記に該当しない写真は、以下の2つに厳密に分類する：
  - **よく似た写真**（採用写真と同じ被写体・ほぼ同じ構図で、追加の情報量がほとんどない）→ `Images/OtherPictures/` に移動。Report.md 末尾の「その他の写真」章に表示する
  - **不採用**（ほぼ同一画角で内容が同じ・明らかなピンぼけ・手ブレで内容不明・誤写）→ `Images/unUsed/` に移動。Report.md には一切表示しない
- 判断基準：「採用済み写真と見た目・被写体がほぼ同じか、または明確に失敗写真か」を問う。どちらでもなければ本文採用
- **写真やレポート内容の配置は、原則としてその日の実際の時系列（撮影時刻順）を優先する。**例えば終日日程なら「朝食 → 訪問風景 → 展示会・企業訪問 → 昼食 → ミーティング → 晩ごはん」のような流れになるが、これはあくまで一例。昼からの半日日程・夜のみの会食など、案件ごとに流れは異なるため、決まったパターンに当てはめず実際のタイムスタンプに従う。多人数撮影の場合は前述「多人数撮影への対応」のとおりGPS情報で場面のまとまりを確認する。そのうえで話の流れに沿って挿入
- 写真1〜2枚＋説明3行以内をセットで配置
- **同じ時間帯に写真が5枚以上ある場合は、およそ半数を2枚横並び（width="390"）で表示し、重要な写真のみ1枚表示にする**
  - 1枚表示に残すもの：場面の切り替わりを示す写真（訪問先の外観など）、地の文の主張を裏付ける決定的な写真、視察・商談の様子がよくわかる代表カット
  - 2枚横並びに回すもの：同じ被写体の接写が続くもの、細部のクローズアップ、似た画角の全景の2枚目以降、補足的なカット
  - 横並びの2枚は必ず**時系列で隣り合う2枚**を組にする（順序を入れ替えない）
  - 縦位置と横位置を組み合わせると見栄えが悪いため、横並びは縦同士・横同士で組む
  - 横並びのキャプションは1行にまとめ、`<p style="color:#888888; font-size:1.05em;">（左）〜　（右）〜</p>` の形式で写真の下に置く。**キャプションは必ず画像ペアの閉じタグ `</p>` の外側（後ろ）に置き、`<p>` ブロックの内側に入れない**（HTML構造が壊れ、PDF変換等でレイアウトが崩れる）
  - 表（table）のセル内に小さな画像（200px等）を並べる形式は見にくいため使わない。写真の比較・列挙にも横並び2枚＋キャプション形式を使う
- 写真の解説は必要・タイトルは不要
- **解説文（キャプション）は、本文の文言をなぞらない。**本文と同じ内容の繰り返しは面白くない。写真ならではの視点（細部・気づき・ユーモア・その場の空気）を一言足す。本文が事実を語り、キャプションが表情を語る、という分担にする
  - 悪い例：本文「PHEVの電源でお湯を沸かした」→ キャプション「PHEVの電源でお湯を沸かした」
  - 良い例：同上 → キャプション「リアゲートの下が、即席のキッチンになる。RAV4は走るベースキャンプである」
  - キャプション芸の型：詠嘆（「あぁ山小屋の風景」）／ツッコミ・疑問形（「USアーミー？」）／真顔の自虐／仲間への温かいいじり／和製英語の遊び（正しさより味）。事実を1つ添えたうえで、これらを一言のせると効く
- 写真の直前には必ず `<br>` を1行入れて上側2行分の余白を確保する
- 写真は `<img src="images/ファイル名" width="800">` で挿入し、解説文は写真の下に配置する
- **縦位置写真（pixelHeight > pixelWidth）は width="500" にする**（縦長になりすぎるため）。`sips -g pixelWidth -g pixelHeight` で確認する。横並び2枚組（width="390"）はそのままでよい
  - **縦横の判定は、目視の印象や目視代行エージェントの報告に頼らず、必ず sips の実寸で全数確認する**（目視による縦横報告は誤りが多いことが実証されている）
- **`width` 属性の値（800／500／400／390）は、実ファイルの画素数とは無関係のレイアウト判別マーカーである。実ファイルを1600pxにリサイズしても、この属性値は絶対に書き換えないこと。** make-pdf.md は `.fig.w800 { width: 100% }` のようにこの値をCSSクラスの分岐に使っており、書き換えると紙面レイアウトが崩れる。実ファイル1600px／属性800 の組み合わせにより、Markdownプレビューでは Retina 2倍相当、PDFではA4幅で約220dpi となる
- 解説文は本文と区別できるよう、グレー色・小さめフォントで表示する：`<p style="color:#888888; font-size:1.05em;">解説文</p>`

## 冒頭の作成日・最終更新日（必須）

README.md の ★/◎/△ バッジは相対時間の自動判定であり、根拠が README.md からは見えない。信憑性を持たせるため、Report.md 本文の**H1タイトルの直後**に作成日・最終更新日を記す。

```markdown
# 展示会名 視察報告

> 作成日：YYYY-MM-DD　最終更新日：YYYY-MM-DD

> 記入者：山崎　/　...
```

- **初稿作成モード（新規作成）**：作成日・最終更新日の両方に本日の日付（`date "+%Y-%m-%d"`）を入れる
- **追記モード（既存Report.mdへの追記）**：既存の `> 作成日：` 行を探し、作成日はそのまま変更せず、最終更新日だけを本日の日付に書き換える。行が見つからない場合のみ新規に挿入する（作成日は不明のため、その場合は追記日を作成日として代用してよい）
- この行は他のスキル（review-report.md・publish-report.md）が触れても壊さないよう、`> 作成日：` で始まる行として独立させる（他のブロック引用行と混在させない）
- 最終更新日は、その後 `publish-report.md` が実行された際に、PUBLISH_SUMMARY.md の実行日時と同じ値に上書きされる（詳細は publish-report.md 側の記載を参照）。README.md のバッジ判定基準（PUBLISH_SUMMARY.md）と Report.md 内の表示を一致させるためである

## 冒頭の改訂履歴（必須）

長期運用のレポートは追記が本文のあちこちに埋まり、あとから読む人が更新箇所へたどり着けない。そのため `> 作成日：` 行の直後（`---` 区切りの前）に改訂履歴の表を置き、各追記セッションへのジャンプリンクを一覧にする。

```markdown
## 📜 改訂履歴

| 日付 | 追記者 | 内容（クリックでジャンプ） |
|---|---|---|
| YYYY-MM-DD | 氏名 | 初版作成 |
| YYYY-MM-DD | 氏名 | [追記内容の見出し](#add-YYYYMMDD-アカウント名) |
```

- **初稿作成モード**：「初版作成」の1行だけの表を作る（初版行にリンクは付けない）
- **追記モード**：追記セッションごとに1行追加する。リンク先は追記マーカー直前に置いたアンカー（前述「ジャンプ用アンカー」参照）。表がない既存レポートの場合は、本文中の既存追記マーカー（`▼ 追記（…）`）を洗い出してアンカーを補い、表を新設する
- 内容欄は追記の見出しをそのまま使い、リンクテキストにする
- 最終更新日と表の最新行の日付がずれないよう、追記時は両方を同時に更新する

## 訪問先企業のWeb情報の収集・掲載ルール

訪問先・視察先の企業については、公式サイト・SNS・業界メディア等の公開情報からその企業を紹介する画像を **2〜6枚** 収集し、レポート本文に **2枚横並び（width="390"）** で掲載する。

- 収集元の優先順位：①企業公式サイト（製品ページ・ニュース・会社紹介）②企業の公式SNSアカウント ③業界メディア・展示会主催者のニュース記事
- 選ぶ画像は、現地で見聞きした内容の**裏付けになるもの**を優先する（生産設備・品質認証・展示会出展・主力製品・ユーザー事例など）。現地写真と同じ内容の重複は避ける
- 掲載位置は「4. 背景（環境変化）」の末尾に `### 公開情報から見る〇〇` の小見出しを立てて配置する
- **キャプションには必ず「何が写っているか」と「出典」を明記する**。出典は媒体名をリンクにし、掲載日（判明する場合）と取得日を添える
  - 例：`（出典：[〇〇公式サイト ニュース](URL)、2026-06-24掲載・2026-07-20取得）`
- 横並び2枚のキャプションは1行にまとめ、`（左）〜（出典：…）　（右）〜（出典：…）` の形式にする
- ファイル名は `会社名_web_内容.jpg` の形式とし、`images/` 直下に置く（現地撮影写真と区別できるようにする）
- ダウンロード後は横幅1600px程度の.jpgに変換する。横幅1200px未満の画像は解像度不足のため採用しない
- 節の末尾に、権利上の注意書きを1行入れる：`※本節の画像は、いずれも〇〇が自社サイト等で公開している広報用素材である。社内報告での参照にとどめ、社外配布資料へ転載する場合は同社の許諾を得ること。`

### ユーザーの生の声の探索

- 訪問先企業の製品・サービスについて、**第三者によるユーザーの声**（レビュー・使用報告・フォーラム投稿等）をSNS・レビューサイト・業界メディア・ECから探索する。輸出企業の場合は輸出先市場の言語（英語・ロシア語・中国語等）まで範囲を広げる
- 実名・出典の確認できる声が見つかった場合は、原文のまま出典付きで引用する（翻訳を添えてよい）
- **見つからなかった場合は、推測・創作で補わない。**代わりに探索範囲と結果を表で記録し、「第三者評価が存在しない」こと自体が持つ意味（市場での実績段階、先行ユーザーの評価を参照できないリスク、現地ヒアリングの希少価値）を本文に書く

## インターネット情報の使用ルール
- Nippou.txt の中から展示会の名称を特定し、インターネットで開催概要（主催者・会期・会場・規模・テーマ等）を調査して、Report.md の冒頭概要セクションに端的に記す
- 展示会の公式ホームページがある場合は、展示会の内容が最も端的にわかるイメージ図（メインビジュアル・キービジュアル等）を1枚 `images/` フォルダにダウンロードし、Report.md の冒頭概要セクションに表示する
  - ダウンロード後、必ず `sips -g pixelWidth` で解像度を確認する
  - 横幅が **1200px 未満** の場合は解像度不足と判断し、別ソース（出展社プレスリリース・業界メディア・Response.jp 等）から同じ展示会のより大きな画像を再検索・再取得する
  - 最終的に横幅 1200px 以上の画像を採用すること。公式バナー（ロゴ＋開催情報入り）と会場全景写真の両方が入手できた場合は、2枚とも掲載する
- ソース（URL・サイト名・取得日）を本文中に明記

## 完了後のアクション
- トップページ（`../../README.md`）の「出張報告書」目次テーブルに行を追加する（レポートは `Reports/202xxx/` 配下なので README.md は2階層上）
- テーブルは **7列構成**（先頭：最終更新列、2列目：更新バッジ列、末尾：ナレッジ化列）。日付順（古い順）に挿入する：

```
| 最終更新 |  | 日付 | 出張先 | 参加者 | 写真枚数・容量 | ナレッジ化 |
|:---:|:---:|---|---|---|:---:|:---:|
```

- 追加する行の形式（README.md からの相対パスを使う）。最終更新・バッジ列は空欄（`—` / 空）、ナレッジ化列は `—` で追加する（最終更新・バッジ列の実際の値は、後述の `StarUpdate.md` が本スキルの最後に自動で埋める。ナレッジ化の日付は archive-report が更新する）：

```
| — |  | 20XX年X月X日 | [🇯🇵 日本／展示会名（会場名）](Reports/フォルダ名/Report.md) | 山崎 | XX枚・X.XMB | — |
```

- 「写真枚数・容量」は、Images フォルダ直下（OtherPictures・unUsed 除く）の採用写真を集計する：

```bash
# 枚数（Images/ 直下のみ。OtherPictures/ と unUsed/ は除外）
find Images -maxdepth 1 -type f \( -iname "*.jpg" -o -iname "*.jpeg" -o -iname "*.png" -o -iname "*.webp" \) | wc -l
# 合計サイズ
find Images -maxdepth 1 -type f \( -iname "*.jpg" -o -iname "*.jpeg" -o -iname "*.png" -o -iname "*.webp" \) \
  -exec stat -f "%z" {} \; | awk '{s+=$1} END {printf "%.1fMB\n", s/1024/1024}'
```

### README.md の最終更新・バッジ表示更新（必須・本スキルの最後に必ず実行する）

README.md に行を追加・変更した場合、**本スキルの最後に必ず `StarUpdate.md`（`.claude/commands/StarUpdate.md`）を実行する**。

- レビュー・公開前チェック・ナレッジ化を待たず、Report.md を作成・追記した時点でトップページの `最終更新` 列・★/◎/△バッジを最新化するためである
- `StarUpdate.md` はREADME.mdの表示（最終更新列・バッジ列）のみを洗い直す専任スキルであり、行の内容（リンク・参加者・写真枚数等）には触れない。行の追加・編集は本スキルの役割のまま変わらない
- `StarUpdate.md` を Read ツールで読み込み、その指示に従って実行すること。「相当の処理」と解釈して独自実装しない
- README.md に変更が一切なかった回（写真・動画審査のみモード等）は実行しなくてよい

## その他の写真コーナー

Report.md の本文（「まとめ」章）の最後に以下の章を追加する。

### 章の位置

```
## 1. サマリー
...
## 6. まとめ
...
---

## その他の写真
```

### 章のフォーマット

```markdown
---

## その他の写真

本編の流れには収めなかった写真・動画サムネイルを収録する。

（写真が存在する場合）
<br>
<p>
<img src="Images/OtherPictures/ファイル名1.jpg" width="390">
<img src="Images/OtherPictures/ファイル名2.jpg" width="390">
</p>

（1枚だけ余る場合は単独で800幅で掲載）
<br>
<img src="Images/OtherPictures/ファイル名.jpg" width="800">

（動画サムネイルが存在する場合は、写真の後に続けて同じフォーマットで掲載する）
```

### ルール

- `Images/OtherPictures/` 内の**全ファイル**を漏れなく掲載する
- `Images/unUsed/` 内のファイルはこの章に**一切表示しない**
- 写真（JPG/PNG）と動画サムネイル（_thumb.jpg）を分けて並べる必要はない。時系列（ファイル名順）で並べてよい
- キャプションは省略可。ただし1行以上のコメントが付けられる場合は `<p style="color:#888888; font-size:1.05em;">` で追加する
- この章は `review-report.md` / `publish-report.md` 実行のたびに OtherPictures の内容と同期して更新する
